# SIGNALS_MODULE — Full Engineering Spec

## Purpose

Define the complete engineering specification for the signals module.
This spec must contain enough detail that a builder agent can implement
the entire module without asking engineering design questions.

## Module Ownership

The signals module owns:

- Signal persistence and lifecycle management
- Signal validation (post-creation checks)
- Signal deduplication
- Signal expiration
- Signal analytics queries
- The signals database table

The signals module does NOT own:

- Signal generation (owned by strategy module)
- Risk evaluation (owned by risk module, consumes signals)
- Order creation (owned by paper trading module)
- Position state (owned by portfolio module)

---

## 1. What a Signal Represents

A signal is a recorded intent from a strategy. It states:
"strategy X wants to do Y with symbol Z at this moment."

A signal is NOT an order. It is NOT a trade. It is the output of
strategy evaluation, frozen in time so it can be reviewed, audited,
and acted upon by downstream modules.

### Pipeline Position

```
Strategy evaluates → Signal created → Risk evaluates → Order created (or rejected)
```

### Three Purposes

1. **Input to risk engine** — risk evaluates and decides approve/reject/modify
2. **Audit trail** — every signal ever generated is recorded, whether or not
   it led to a trade
3. **Analytics** — signal frequency, hit rate, strategy effectiveness

---

## 2. Signal Data Model

```
Signal:
  - id: UUID
  - strategy_id: UUID (FK → Strategy)
  - strategy_version: str
  - symbol: str
  - market: str (equities | forex)
  - timeframe: str
  - side: str (buy | sell)
  - signal_type: str (entry | exit | scale_in | scale_out)
  - source: str (strategy | manual | safety | system)
  - confidence: float (0.0 to 1.0, nullable for non-strategy sources)
  - status: str (pending | risk_approved | risk_rejected | risk_modified |
                  expired | canceled)
  - payload_json: dict (strategy reasoning, indicator values at signal time)
  - position_id: UUID, nullable (FK → Position, for exit signals)
  - exit_reason: str, nullable (stop_loss | take_profit | trailing_stop |
                                 max_hold | condition | manual | safety | system)
  - ts: datetime (evaluation timestamp when signal was generated, UTC)
  - expires_at: datetime, nullable (UTC)
  - created_at: datetime (UTC)
  - updated_at: datetime (UTC)

Indexes:
  INDEX (strategy_id, created_at)
  INDEX (symbol, created_at)
  INDEX (status)
  INDEX (strategy_id, symbol, status)
```

### Field Details

**strategy_version:** The exact config version the strategy was running
when this signal was generated. Enables tracing any signal back to the
exact config that produced it.

**confidence:** Optional score from the strategy indicating signal strength.
For config-driven strategies, defaults to 1.0. More meaningful for custom
strategies with nuanced scoring. The risk engine can use this as input.

**payload_json:** Auto-populated by the strategy runner with indicator
values at signal time. This is the strategy's explanation of why it signaled:

```json
{
  "indicators": {
    "rsi_14": 28.5,
    "ema_200": 185.10,
    "adx_14": 31.2,
    "volume": 872000
  },
  "conditions_met": [
    "rsi(14) < 70: true (28.5)",
    "rsi(14) > 30: true (28.5)",
    "close crosses_above ema(200): true",
    "adx(14) > 25: true (31.2)",
    "volume > 500000: true (872000)"
  ],
  "close_price": 186.42
}
```

**position_id:** For exit signals, links to the specific position being closed.
Entry signals have this as null. Prevents ambiguity when a strategy has
multiple positions.

**exit_reason:** Distinguishes why an exit was triggered. Feeds analytics:
"70% of exits are stop losses" reveals entry timing issues.

**source values:**
- "strategy" — generated by strategy evaluation (condition engine)
- "manual" — user-initiated through the UI (close position button)
- "safety" — generated by the safety monitor for orphaned positions
- "system" — generated by kill switch or auto-close logic

**expires_at:** Computed at creation time based on strategy timeframe.
Signals not processed before this time are marked expired.

## 3. Signal Validation

When the strategy module creates a signal and passes it to the signals
module for persistence, validation occurs before saving.

### Required Field Checks

- strategy_id exists and references a valid, non-deleted strategy
- symbol is on the active watchlist
- side is "buy" or "sell"
- signal_type is one of: entry, exit, scale_in, scale_out
- source is one of: strategy, manual, safety, system
- ts is not in the future
- ts is not more than 5 minutes in the past (stale signal guard)
- payload_json is valid JSON (if provided)

### Logical Checks

- **Entry signal:** No existing open position for this strategy + symbol
  (unless strategy config allows multiple positions via max_positions)
- **Exit signal:** An open position exists for this strategy + symbol
- **Exit signal:** position_id references a real position owned by this strategy
- **Scale_in signal:** Position exists and is not at max configured size
- **Scale_out signal:** Position exists and has quantity to reduce

### On Validation Failure

- Signal is NOT persisted
- A validation failure log is written with details
- The strategy evaluation still counts as successful (strategy did its job,
  signal failed downstream validation)
- No error thrown back to strategy runner

## 4. Signal Deduplication

### The Problem

Without deduplication, a strategy can emit the same signal repeatedly
as long as conditions remain true. RSI stays below 70, EMA crossover
persists — every evaluation cycle produces another identical BUY signal,
each potentially creating a new order.

### Dedup Rules

A signal is considered a duplicate if ALL of the following match an
existing signal with status = "pending" or "risk_approved":

- Same strategy_id
- Same symbol
- Same side
- Same signal_type
- Existing signal is less than SIGNAL_DEDUP_WINDOW_BARS old
  (measured in the strategy's timeframe)

### On Duplicate Detection

- Do NOT create a new signal
- Log: "Signal deduplicated: BUY AAPL from rsi_ema_momentum
  (duplicate of signal {id} from {timestamp})"
- The existing signal remains in the pipeline unchanged

### Dedup Window

Configurable per strategy (with global default fallback):

```
Strategy config:
  signal_dedup_window_bars: int (default uses global setting)

Global default:
  SIGNAL_DEDUP_WINDOW_BARS=1
```

- Window of 1: ignore duplicate within 1 bar of the same timeframe
- Window of 0: no dedup, allow every signal (for scale-in strategies
  that intentionally add on every bar)

### Dedup Exceptions

- **Exit signals are NOT deduplicated.** If the strategy says exit,
  it means exit now. Multiple exit signals reinforce urgency.
- **Manual signals are NOT deduplicated.** User intent is always honored.
- **Safety monitor signals are NOT deduplicated.** Safety exits are critical.
- **System signals are NOT deduplicated.** Kill switch signals are emergency.

Only source="strategy" entry and scale_in signals are subject to dedup.

## 5. Signal Lifecycle

### States

```
pending → risk_approved → (consumed by paper trading engine)
       → risk_rejected
       → risk_modified → (consumed with modifications)
       → expired
       → canceled
```

**Pending:** Initial state. Signal created, awaiting risk evaluation.

**Risk Approved:** Risk engine approved. Paper trading engine will create
an order from it. Terminal state for the signals module (downstream
modules take over).

**Risk Rejected:** Risk engine rejected. Recorded with reason in
risk_decisions table. No order created. Terminal state.

**Risk Modified:** Risk engine approved with modifications (e.g., reduced
quantity, adjusted parameters). Modified version is what gets executed.
Terminal state for signals module.

**Expired:** Signal was pending too long without risk evaluation.
Guards against pipeline stalls. Terminal state.

**Canceled:** Signal was superseded — strategy disabled, user manually
closed position, or a newer signal replaced it. Terminal state.

### Transition Rules

All transitions are one-directional. A rejected signal cannot be re-approved.
A new signal must be generated.

```
pending → risk_approved:  risk engine approves
pending → risk_rejected:  risk engine rejects
pending → risk_modified:  risk engine approves with changes
pending → expired:        expiry checker finds signal past TTL
pending → canceled:       strategy disabled, position manually closed,
                          or signal explicitly canceled via API
```

No other transitions are valid.

## 6. Signal Expiration

### Purpose

Prevent stale signals from being acted upon. A signal that sits in
"pending" too long reflects a market state that no longer exists.

### Expiry Calculation

```
Signal.expires_at = Signal.ts + expiry_duration
```

Expiry duration is derived from the strategy's timeframe:

```
1m strategy:   expiry = 2 minutes
5m strategy:   expiry = 10 minutes
15m strategy:  expiry = 30 minutes
1h strategy:   expiry = 60 minutes
4h strategy:   expiry = 4 hours
1d strategy:   expiry = 24 hours
```

Can be overridden in strategy config or globally:

```
SIGNAL_EXPIRY_SECONDS=300  (global default fallback: 5 minutes)
```

### Expiry Background Job

Runs every SIGNAL_EXPIRY_CHECK_INTERVAL_SEC (default: 60 seconds):

```sql
UPDATE signals
SET status = 'expired', updated_at = NOW()
WHERE status = 'pending'
  AND expires_at < NOW()
```

Expired signals are logged. Consistent expiration indicates the risk
engine is not keeping up — this is a health concern that should surface
in the observability layer.

## 7. Signal-to-Risk Handoff Contract

The interface between signals and risk is explicit. The risk engine
consumes signals in "pending" status.

### What risk reads from signals

```
signal.id
signal.strategy_id
signal.strategy_version
signal.symbol
signal.market
signal.side
signal.signal_type
signal.source
signal.confidence
signal.payload_json
signal.position_id (for exit signals)
```

### What risk writes back

```
signal.status → risk_approved | risk_rejected | risk_modified
signal.updated_at → current timestamp
```

### What risk creates separately

```
RiskDecision:
  - id: UUID
  - signal_id: UUID (FK → Signal)
  - status: str (approved | rejected | modified)
  - reason_code: str
  - reason_text: str
  - modifications_json: dict, nullable (if modified: what changed)
  - ts: datetime
  - created_at: datetime
```

The risk engine does NOT modify signal fields beyond status and updated_at.
All risk-specific data goes into the RiskDecision table. The signal record
remains a clean, immutable record of what the strategy intended.

## 8. Signal Analytics

### Supported Queries

The signals table supports strategy performance analysis:

**Signal frequency per strategy:**
"How many signals did strategy X emit this week?"

```sql
SELECT COUNT(*) FROM signals
WHERE strategy_id = :id
  AND created_at >= :week_start
```

**Approval rate:**
"What percentage of signals were approved by risk?"

```sql
SELECT status, COUNT(*) FROM signals
WHERE strategy_id = :id
GROUP BY status
```

**Win rate:**
"Of approved signals that became trades, what percentage were profitable?"
(Requires join with paper_fills and positions — cross-module query)

**Signal timing:**
"What time of day does strategy X generate the most signals?"

```sql
SELECT EXTRACT(HOUR FROM ts) as hour, COUNT(*) FROM signals
WHERE strategy_id = :id
GROUP BY hour
ORDER BY count DESC
```

**Rejection analysis:**
"Why are signals getting rejected?"
(Requires join with risk_decisions)

```sql
SELECT rd.reason_code, COUNT(*) FROM risk_decisions rd
JOIN signals s ON s.id = rd.signal_id
WHERE s.strategy_id = :id AND rd.status = 'rejected'
GROUP BY rd.reason_code
```

**Exit reason distribution:**
"How are positions being closed?"

```sql
SELECT exit_reason, COUNT(*) FROM signals
WHERE strategy_id = :id AND signal_type = 'exit'
GROUP BY exit_reason
```

These queries run against existing tables with the defined indexes.
No special analytics tables needed for MVP.

## 9. Signal Notifications

When signals are created or change status, the system can notify
the user through the dashboard activity feed.

### Notification Events

```
signal.created     → "Strategy X: BUY AAPL signal generated"
signal.approved    → "Signal approved by risk: BUY AAPL"
signal.rejected    → "Signal rejected: BUY AAPL (reason: exposure limit)"
signal.modified    → "Signal modified: BUY AAPL qty reduced from 100 to 50"
signal.expired     → "Signal expired: BUY AAPL (pending too long)"
signal.canceled    → "Signal canceled: BUY AAPL (strategy disabled)"
```

For MVP, these appear in the dashboard activity feed and the signals view.
Future enhancements: email, webhook, push notifications (configurable
per strategy or globally).

```
backend/app/signals/
    __init__.py
    service.py              ← signal creation, validation, dedup, lifecycle
    models.py               ← SQLAlchemy models
    schemas.py              ← Pydantic request/response schemas
    router.py               ← API endpoints
    config.py               ← signal module configuration

```

This is a thin module. Complexity lives in strategies (generation)
and risk (evaluation). Signals is the clean handoff layer between them.

## 11. API Endpoints

```
GET  /api/v1/signals                  → list signals with filters
                                        query params: strategy_id, symbol,
                                        status, signal_type, source,
                                        date_start, date_end,
                                        page, page_size
GET  /api/v1/signals/:id              → signal detail with full payload
GET  /api/v1/signals/recent           → last N signals across all strategies
                                        query params: limit (default 20)
GET  /api/v1/signals/stats            → signal analytics summary
                                        query params: strategy_id (optional),
                                        date_start, date_end
                                        returns: count by status, by strategy,
                                        by symbol, by signal_type
POST /api/v1/signals/:id/cancel       → cancel a pending signal
                                        only valid for status = "pending"
```

Signals are NOT created through the API directly. They are created by:
- Strategy runner (source: strategy)
- Manual close actions (source: manual)
- Safety monitor (source: safety)
- Kill switch / system actions (source: system)

The signals API is read-only plus cancel.
